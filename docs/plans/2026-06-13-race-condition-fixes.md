# Race-Condition Fixes Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Close the concurrency bugs found in the 2026-06-13 race-condition audit ã eliminate the price-target double-send, the STOP-vs-verify re-opt-in, the `short_urls` dedup race, and the non-atomic Grok counter; add per-channel retry for instant price alerts; and document the accepted reserved-slot at-least-once tradeoff.

**Architecture:** Push correctness into the database layer (atomic `UPDATE ãÎ RETURNING` / `ON CONFLICT` / guarded `WHERE`) rather than application-level check-then-act, matching the codebase's existing claim-RPC pattern (`claim_scheduled_notification`, `reserve_*`/`finalize_*`). Per-channel price-alert retry persists `email_delivered`/`sms_delivered` flags on the cooldown row so a failed channel is re-attempted on a subsequent eligible tick without re-sending the channel that already succeeded.

**Tech Stack:** Astro 5 SSR, Supabase (Postgres + plpgsql RPCs), TypeScript, Vitest (real Supabase for integration tests, mocked Supabase for unit tests), Biome.

## Spec (source: in-conversation audit, 2026-06-13)

Findings to address, by severity:

1. **ÞÇ Price-target double-send** ã `src/lib/price-targets/process.ts:247-267`. The "mark pending" CAS `update().is("triggered_at", null)` never checks the affected row count (no `.select()`), so two overlapping scheduler invocations both deliver. **Fix:** check rows returned; skip when zero.
2. **ÞÀ STOP-vs-OTP-verify re-opt-in** ã `src/pages/api/auth/sms/verify-code.ts:169-174`. Verify computes `sms_notifications_enabled` from a stale read; a concurrent inbound STOP can be clobbered, re-enabling SMS for a user who just opted out (TCPA-adjacent). **Fix:** guard the re-enable with a DB-side `WHERE sms_opted_out = false`.
3. **ÞÀ `short_urls` dedup race** ã `src/lib/messaging/sms/url-shortener.ts:48-89`. Check-then-act with no unique constraint on `original_url`; concurrent calls insert duplicate rows for the same URL. **Fix:** `UNIQUE(original_url)` + atomic get-or-create RPC.
4. **ÞÂ Non-atomic Grok counter** ã `src/lib/daily-digest/process.ts:225-245` and the duplicated copy in `src/lib/staged-notifications/deliver.ts:594-623`. Read-modify-write on `users.grok_sends_in_window` loses increments. **Fix:** one atomic RPC doing reset-or-increment in SQL; both call sites use it.
5. **Per-channel price-alert retry** (product decision: *add it*) ã `src/lib/market-notifications/delivery.ts:229` finalizes the cooldown slot on any single-channel success, silently dropping the other channel for the trading day. **Fix:** persist per-channel delivery flags; finalize only when all enabled channels delivered, else mark `partial` and retry the missing channel on the next eligible tick.
6. **Reserved-slot 10-min TTL at-least-once** (product decision: *document only*) ã bounded duplicate-send if a worker crashes after send, before finalize. **Fix:** documentation + code comment, no behavior change.

**Acceptance:** every fix has a failing-first test that pins the concurrent-safe behavior; `npm run check:biome`, `npm run check:ts`, `npm test`, and `npm run check:db-privileges` pass; each new migration bumps `app_metadata.schema_version` and `EXPECTED_DB_SCHEMA_VERSION` in lockstep.

**Migration-timestamp note:** migration filenames are generated by `supabase migration new <name>` at run time and cannot be predicted. Where a step writes a migration, run the command first, then edit the generated file and set both its `UPDATE app_metadata ãÎ SET value = '<generated filename without .sql>'` and `tests/helpers/constants.ts`'s `EXPECTED_DB_SCHEMA_VERSION` to that exact generated value. Because the CLI always issues a later timestamp, the most-recently-created migration is the one both fields must reference after each migration task.

```sql
-- Atomic short-URL dedup: enforce one row per original_url and provide an
-- atomic get-or-create RPC, closing the check-then-act race in url-shortener.ts.

SET lock_timeout = '5s';
SET statement_timeout = '30s';

-- Collapse any pre-existing duplicates (keep the row with the latest expiry).
DELETE FROM public.short_urls s
USING public.short_urls keep
WHERE s.original_url = keep.original_url
  AND s.expires_at < keep.expires_at;
-- Tie-break remaining exact-expiry duplicates by id.
DELETE FROM public.short_urls s
USING public.short_urls keep
WHERE s.original_url = keep.original_url
  AND s.expires_at = keep.expires_at
  AND s.id > keep.id;

DROP INDEX IF EXISTS public.idx_short_urls_original_url;
ALTER TABLE public.short_urls
  ADD CONSTRAINT short_urls_original_url_key UNIQUE (original_url);

CREATE OR REPLACE FUNCTION public.get_or_create_short_url(
  p_id text,
  p_original_url text,
  p_expires_at timestamptz
) RETURNS text
  LANGUAGE plpgsql
  SET search_path TO public, pg_temp
AS $$
DECLARE
  v_id text;
BEGIN
  INSERT INTO public.short_urls (id, original_url, expires_at)
  VALUES (p_id, p_original_url, p_expires_at)
  ON CONFLICT (original_url) DO UPDATE
    SET expires_at = GREATEST(public.short_urls.expires_at, excluded.expires_at)
  RETURNING id INTO v_id;

  RETURN v_id;
END;
$$;

REVOKE ALL ON FUNCTION public.get_or_create_short_url(text, text, timestamptz) FROM PUBLIC;
GRANT EXECUTE ON FUNCTION public.get_or_create_short_url(text, text, timestamptz) TO service_role;

-- DO UPDATE on conflict requires UPDATE on the table; service_role currently
-- has only INSERT/SELECT/DELETE.
GRANT UPDATE ON TABLE public.short_urls TO service_role;

UPDATE public.app_metadata
SET value = '<GENERATED_TS>_short_urls_unique_original_url'
WHERE key = 'schema_version';
```

  // conflict the RPC refreshes expiry and returns the existing id. We still

- [ ] **Step 1: Create the migration**

Run: `supabase migration new atomic_grok_counter`

```sql
-- Atomic Grok rolling-window counter: reset-or-increment in one statement so
-- concurrent deliveries for the same user cannot lose an increment.

SET lock_timeout = '5s';
SET statement_timeout = '30s';

CREATE OR REPLACE FUNCTION public.increment_grok_send_counter(
  p_user_id uuid,
  p_now timestamptz,
  p_window_hours integer
) RETURNS integer
  LANGUAGE plpgsql
  SET search_path TO public, pg_temp
AS $$
DECLARE
  v_new_count integer;
BEGIN
  UPDATE public.users u
  SET
    last_grok_rumors_at = p_now,
    grok_window_start = CASE
      WHEN u.grok_window_start IS NULL
        OR u.grok_window_start <= p_now - make_interval(hours => p_window_hours)
      THEN p_now
      ELSE u.grok_window_start
    END,
    grok_sends_in_window = CASE
      WHEN u.grok_window_start IS NULL
        OR u.grok_window_start <= p_now - make_interval(hours => p_window_hours)
      THEN 1
      ELSE u.grok_sends_in_window + 1
    END
  WHERE u.id = p_user_id
  RETURNING u.grok_sends_in_window INTO v_new_count;

  RETURN v_new_count;
END;
$$;

REVOKE ALL ON FUNCTION public.increment_grok_send_counter(uuid, timestamptz, integer) FROM PUBLIC;
GRANT EXECUTE ON FUNCTION public.increment_grok_send_counter(uuid, timestamptz, integer) TO service_role;

UPDATE public.app_metadata
SET value = '<GENERATED_TS>_atomic_grok_counter'
WHERE key = 'schema_version';
```

## Task 5: Per-channel retry for instant price alerts

**Design:** Persist `email_delivered` / `sms_delivered` on `market_asset_price_alert_cooldowns` and a `delivery_status = 'partial'` state. The reserve RPC re-grants a `partial` slot on the same trading day (keeping the per-channel flags) so a subsequent eligible tick retries only the channel that has not yet succeeded; a new trading day resets the flags. `deliverPriceAlert` skips already-delivered channels and reports a per-channel result. The caller finalizes only when every *enabled* channel is delivered, marks `partial` on mixed outcome, and releases when nothing delivered. **Retry is bounded to ticks where the user remains anomaly-eligible** ã acceptable because an instant alert's content is only valid while the move persists, and re-sending stale "unusual move" content later would be wrong. This bound is the key tradeoff; note it in the migration comment.

-- Per-channel delivery for instant price alerts. A slot finalizes only when
-- all enabled channels deliver; a mixed outcome becomes 'partial' and the
-- missing channel retries on the next eligible tick (bounded to ticks where
-- the user remains anomaly-eligible ã instant-alert content is only valid
-- while the move persists, so we deliberately do not retry after it settles).

SET lock_timeout = '5s';
SET statement_timeout = '30s';

ALTER TABLE public.market_asset_price_alert_cooldowns
  ADD COLUMN IF NOT EXISTS email_delivered boolean NOT NULL DEFAULT false,
  ADD COLUMN IF NOT EXISTS sms_delivered boolean NOT NULL DEFAULT false;

-- Re-grant 'reserved' (new day / stale) OR 'partial' (same day) while
-- preserving per-channel flags within a trading day and resetting on a new day.
CREATE OR REPLACE FUNCTION public.reserve_market_asset_price_alert_slot(
  p_user_id uuid,
  p_symbol text,
  p_abs_move_percent numeric DEFAULT 0,
  p_abs_move_dollar numeric DEFAULT 0
) RETURNS boolean
  LANGUAGE plpgsql
  SET search_path TO public, pg_temp
AS $$
DECLARE
  reserved boolean;
  claim_local_ts timestamp;
  claim_trading_day date;
  v_abs_move_percent numeric;
  v_abs_move_dollar numeric;
BEGIN
  v_abs_move_percent := GREATEST(COALESCE(p_abs_move_percent, 0), 0);
  v_abs_move_dollar := GREATEST(COALESCE(p_abs_move_dollar, 0), 0);

  claim_local_ts := now() AT TIME ZONE 'America/New_York';
  claim_trading_day := claim_local_ts::date;
  IF claim_local_ts::time >= time '16:00:00' THEN
    claim_trading_day := claim_trading_day + 1;
  END IF;

  INSERT INTO public.market_asset_price_alert_cooldowns (
    user_id, symbol, last_alerted_at, trading_day_key, alerts_sent_count,
    max_abs_move_percent, max_abs_move_dollar, delivery_status, reserved_at,
    email_delivered, sms_delivered
  )
  VALUES (
    p_user_id, p_symbol, now(), claim_trading_day, 0,
    v_abs_move_percent, v_abs_move_dollar, 'reserved', now(), false, false
  )
  ON CONFLICT (user_id, symbol) DO UPDATE
    SET
      delivery_status = 'reserved',
      reserved_at = now(),
      trading_day_key = claim_trading_day,
      max_abs_move_percent = v_abs_move_percent,
      max_abs_move_dollar = v_abs_move_dollar,
      email_delivered = CASE
        WHEN public.market_asset_price_alert_cooldowns.trading_day_key < claim_trading_day
        THEN false ELSE public.market_asset_price_alert_cooldowns.email_delivered END,
      sms_delivered = CASE
        WHEN public.market_asset_price_alert_cooldowns.trading_day_key < claim_trading_day
        THEN false ELSE public.market_asset_price_alert_cooldowns.sms_delivered END
    WHERE
      public.market_asset_price_alert_cooldowns.trading_day_key < claim_trading_day
      OR public.market_asset_price_alert_cooldowns.delivery_status = 'partial'
      OR (
        public.market_asset_price_alert_cooldowns.delivery_status = 'reserved'
        AND public.market_asset_price_alert_cooldowns.reserved_at < now() - interval '10 minutes'
      )
  RETURNING true INTO reserved;

  RETURN COALESCE(reserved, false);
END;
$$;

-- Finalize from 'reserved' OR 'partial'; mark both channels delivered.
CREATE OR REPLACE FUNCTION public.finalize_market_asset_price_alert_slot(
  p_user_id uuid,
  p_symbol text
) RETURNS boolean
  LANGUAGE plpgsql
  SET search_path TO public, pg_temp
AS $$
DECLARE
  finalized boolean;
BEGIN
  UPDATE public.market_asset_price_alert_cooldowns
  SET
    delivery_status = 'finalized',
    last_alerted_at = now(),
    reserved_at = NULL,
    alerts_sent_count = 1,
    email_delivered = true,
    sms_delivered = true
  WHERE user_id = p_user_id
    AND symbol = p_symbol
    AND delivery_status IN ('reserved', 'partial')
  RETURNING true INTO finalized;

  RETURN COALESCE(finalized, false);
END;
$$;

-- Mark a mixed-outcome slot 'partial', recording which channels succeeded so a
-- later eligible tick retries only the missing one.
CREATE OR REPLACE FUNCTION public.mark_market_asset_price_alert_slot_partial(
  p_user_id uuid,
  p_symbol text,
  p_email_delivered boolean,
  p_sms_delivered boolean
) RETURNS boolean
  LANGUAGE plpgsql
  SET search_path TO public, pg_temp
AS $$
DECLARE
  marked boolean;
BEGIN
  UPDATE public.market_asset_price_alert_cooldowns
  SET
    delivery_status = 'partial',
    reserved_at = now(),
    email_delivered = p_email_delivered,
    sms_delivered = p_sms_delivered
  WHERE user_id = p_user_id
    AND symbol = p_symbol
    AND delivery_status IN ('reserved', 'partial')
  RETURNING true INTO marked;

  RETURN COALESCE(marked, false);
END;
$$;

REVOKE ALL ON FUNCTION public.reserve_market_asset_price_alert_slot(uuid, text, numeric, numeric) FROM PUBLIC;
REVOKE ALL ON FUNCTION public.finalize_market_asset_price_alert_slot(uuid, text) FROM PUBLIC;
REVOKE ALL ON FUNCTION public.mark_market_asset_price_alert_slot_partial(uuid, text, boolean, boolean) FROM PUBLIC;
GRANT EXECUTE ON FUNCTION public.reserve_market_asset_price_alert_slot(uuid, text, numeric, numeric) TO service_role;
GRANT EXECUTE ON FUNCTION public.finalize_market_asset_price_alert_slot(uuid, text) TO service_role;
GRANT EXECUTE ON FUNCTION public.mark_market_asset_price_alert_slot_partial(uuid, text, boolean, boolean) TO service_role;

UPDATE public.app_metadata
SET value = '<GENERATED_TS>_price_alert_per_channel'
WHERE key = 'schema_version';

### Reserved-slot at-least-once (accepted tradeoff)

`reserve_market_asset_price_alert_slot` re-grants a slot whose `delivery_status = 'reserved'`

in production, this crash-after-send-before-finalize window is the first place to look.
